<objective>
Build the phase-based game state machine with Zustand and wire sessionStorage persistence so that refreshing the browser restores the exact game state, and pressing the browser back button does not destroy or reset the game. This is the architectural core that all game screens in Phase 2 will plug into.

Purpose: The two highest-cost pitfalls in a single-page game app are (1) losing state on refresh and (2) the back button destroying the game. Solving both now means Phase 2 can focus purely on UI/UX without worrying about state architecture.

Output: A Zustand store with typed game phases, sessionStorage sync, hydration guard, and back-button interception — all wired into the app shell with visual proof (phase name displayed).
</objective>

  <name>Task 1: Create Zustand game store with state machine and sessionStorage persistence</name>
  <files>

  <action>
    1. **Create storage utilities** (`src/lib/storage.ts`):
       - `STORAGE_KEY = 'playground-hb-game'`
       - `saveGameState(state: PersistedGameState): void` — serializes to sessionStorage (NOT localStorage — per user decision, state dies with tab close)
       - `loadGameState(): PersistedGameState | null` — deserializes from sessionStorage with try/catch (returns null on parse error or SSR)
       - `clearGameState(): void` — removes the key from sessionStorage
       - Type `PersistedGameState` includes only serializable fields: `phase`, `players`, `currentHole`, `scores` — NO functions, NO transient UI state
       - SSR guard: check `typeof window !== 'undefined'` before any sessionStorage access

    2. **Create Zustand game store** (`src/lib/game-store.ts`):
       - **Game phases** (type-safe union): `'landing' | 'setup' | 'scoring' | 'results'`
         - Note: prototype used `'setup' | 'scoring' | 'leaderboard'` — the new design adds a `'landing'` phase and renames leaderboard to `'results'` per CONTEXT.md
       - **State shape:**
         ```typescript
         interface GameState {
           // Phase
           phase: GamePhase
           // Players
           players: Player[]
           // Scoring
           currentHole: number  // 1-9
           scores: Record<string, number[]>  // playerId -> [hole1Score, hole2Score, ...]
           // Hydration
           hydrated: boolean
         }
         ```
       - **Player type:** `{ id: string; name: string }` — scores stored separately in `scores` map (cleaner than embedding in player object)
       - **Actions (state machine transitions):**
         - `goToSetup()` — only from 'landing', sets phase to 'setup'
         - `goToScoring()` — only from 'setup', validates players exist and have names, initializes scores map with empty arrays, sets currentHole to 1, sets phase to 'scoring'
         - `goToResults()` — only from 'scoring', sets phase to 'results'
         - `goToLanding()` — from 'results' only (play again), clears all game data, sets phase to 'landing'
         - Each transition guard: if current phase is wrong, log warning and return (do not throw)
       - **Scoring actions:**
         - `setScore(playerId: string, hole: number, score: number)` — updates scores map
         - `nextHole()` — increments currentHole (max 9)
         - `prevHole()` — decrements currentHole (min 1) — not in v1 UI but nice for store completeness
       - **Player actions:**
         - `addPlayer(name: string)` — adds player with generated ID
         - `removePlayer(id: string)` — removes player (must keep at least 0; UI enforces minimum)
         - `updatePlayerName(id: string, name: string)` — updates name
       - **Hydration:**
         - `hydrate()` — called once on mount. Reads from sessionStorage via `loadGameState()`. If state found, merges into store. Sets `hydrated = true` regardless.
         - `hydrated` flag starts `false` — components should show nothing (or a loading skeleton) until `hydrated === true` to prevent flash of default state
       - **Persistence subscription:**
         - Use Zustand's `subscribe` to auto-save to sessionStorage on every state change (after hydration). Only persist the serializable subset (phase, players, currentHole, scores). Debounce is unnecessary — sessionStorage writes are synchronous and fast.
         - Alternative: use Zustand persist middleware with `storage: sessionStorage` — either approach works, pick whichever is simpler. If using persist middleware, configure `partialize` to exclude `hydrated` and action functions.

    3. **State machine integrity:**
       - Transitions are GUARDED. Calling `goToScoring()` from `'landing'` does nothing (must go through setup first).
       - This prevents URL manipulation or stale callbacks from corrupting game state.
       - The valid transition graph:
         ```
         landing -> setup -> scoring -> results -> landing (cycle)
         ```
